Over the lifetime of the blog, I tried to improve the discoverability of this site:

 - I first started with classical [search engine optimization](blog/19/) and integration of [webmentions](https://indieweb.org/Webmention).
 - most recently I enabled a simpler commenting system for which the blog post is still missing.
 
However, it is really clear that I avoided the integration with the social networks. I am just not into the daily posts on those networks and hence I get absolutly no traffic from over there. 
Therefore, I came up with a little AI assistent that proposes me ten drafts, I once I accept and edit them, he neatly publishes them once a week. As a result I now have more consistent posts, little AI slop
because I always edit the posts and actually a good variety of summaries on the different content on the page. All of this without a subscription, without an autonomous agent making questionable posts while I sleep, and without a complicated setup.

The result: a little Python agent running on Scaleway for about 20 cents a month.

## The Architecture

The agent is a [LangGraph](https://langchain-ai.github.io/langgraph/) pipeline with three nodes that run sequentially every morning at 08:00 UTC:

1. **Plan** — Selects which blog articles to promote and schedules draft slots across channels
2. **Drafts** — Generates platform-specific post text (Mastodon: up to 500 characters with hashtags; Bluesky: up to 300 characters, no hashtags), then self-critiques and refines each draft once
3. **Publish** — Posts any approved drafts whose scheduled time has passed

The candidate pool of articles comes from Umami analytics — which pages have had recent traffic, which topics are underrepresented in recent posts — but this data is refreshed periodically by hand, not on every run. I pull it when I want to recalibrate what the agent focuses on.

All state lives in a handful of JSON files in S3. There's no database, no message queue, no always-on server. The container spins up, does its work in about two minutes, and shuts down. Why a container and not a serverless function — more on that below.

The full source is in the `growth-agent/` folder of my [GitHub repository](https://github.com/fretchen/fretchen.github.io).

## The Approval Interface

The agent doesn't post automatically. After generating drafts, it drops them into a queue. I review them via a small React page on this site — I can edit the text, approve or reject each draft, and optionally set a specific publish time.

This was a deliberate design choice from the start. The agent's job is to get 80% of the way there: a usable draft with the right hashtags, a reasonable hook, within the character limit. My job is the remaining 20%: is this the right tone for this week? Does it make sense given what else is happening? Is the framing honest?

Treating this as a collaborative tool rather than an autonomous one turns out to matter quite a bit — more on that below.

## Key Learnings

After running this for several months and seeing 46 posts go out across Mastodon and Bluesky, here's what I actually learned.

### Don't hand strategy to the agent

Early versions tried to be smart about strategy: the LLM would analyze traffic data and decide which pages to promote based on pageviews, referrers, and content gaps. In theory this sounds right. In practice, it collapsed onto a narrow set of topics within a few weeks. The analytics-driven planner kept recommending the same three or four high-traffic pages because those had the clearest signal.

The fix was simple: I moved strategy back to myself. The agent now executes within a framework I define — which topics to cover, which channels, how frequently — rather than inferring it from data. The LLM is still responsible for drafting. That's where it genuinely helps.

### Random selection beats algorithmic when you have no data

Related to the above: when there's no engagement history, a "smart" selection algorithm is just noise dressed up as signal. I had no meaningful data about which posts perform better on Mastodon vs. Bluesky, which topics get more shares, or what time of day matters.

The current approach picks from a broader candidate pool with some randomization. It's less elegant but more honest about what we actually know. Premature optimization on engagement signals that don't yet exist is just a way of building a feedback loop on noise.

Once there's enough data — follower engagement, link click rates, referral traffic — it will make sense to introduce smarter ranking. Not yet.

### Think assistant, not autonomous system

The approval step isn't a workaround for an agent that isn't ready to post on its own. It's the intended behavior.

There's an important difference between "the agent posts for me" and "the agent drafts for me." The first requires trusting the agent's judgment on tone, timing, and context. The second just requires trusting that it can produce a reasonable starting point faster than I can write from scratch. The second threshold is much easier to meet with current LLM quality, and it's the one I actually care about.

Running it this way also makes mistakes cheap: if the agent drafts something off-tone or factually vague, I reject it. Nothing goes out without a human read.

### Python agents don't fit serverless — use a container

My other backend services run as Scaleway serverless functions: the image generation API, the growth draft API, the payment facilitator. All TypeScript, all stateless, all fast to start. Naturally I started building the growth agent the same way.

It didn't work. The problem isn't Python itself — it's that a Python AI agent has a large dependency tree: LangGraph, Pydantic, httpx, boto3, a model client. Packaging all of that into a serverless function zip means cold starts that take tens of seconds, package size limits you quickly run into, and a runtime environment that's harder to control. For a TypeScript Lambda importing three packages, serverless is genuinely simple. For a Python LLM agent, it fights you.

The fix was to switch to a container. The agent now runs as a Docker image deployed on Scaleway Container, managed with [OpenTofu](https://opentofu.org/) (the open-source Terraform fork). The infrastructure lives in `growth-agent/terraform/` and defines the container registry, deployment, cron trigger, and environment variable references — all as code. The tradeoff is a bit more setup upfront, but a fully reproducible deployment and no fight with packaging constraints.

If you're building a Python AI agent for periodic tasks: skip serverless functions from the start and go straight to a container with a cron trigger. The runtime predictability is worth it.

### It costs almost nothing

Monthly costs break down roughly like this:

- **LLM API** (IONOS, Llama 3.3 70B): ~15 cents
- **Scaleway container** (runs ~2 min/day): ~5 cents
- **S3 storage** (a few KB/day of JSON): negligible

Total: **~20 cents/month**.

No subscription tier, no per-seat pricing, no usage-based cost that scales uncomfortably with success. The whole thing fits comfortably within serverless free tiers.

## What the Numbers Actually Look Like

To be honest about impact: 46 posts later, the social presence exists but has not visibly moved the needle on website traffic. Monthly visitors: February 118, March 257, April 135, May 169. The March spike happened, but I can't attribute it to the agent — it coincided with other activity and the numbers dropped back in April.

There are reactions on social posts. People see them, occasionally engage, sometimes follow. But "measurable traffic growth from social media" is not something I can claim yet.

This is fine. The goal was a consistent social presence with minimal effort, not an immediate distribution solution. 46 posts went out that wouldn't have existed otherwise. That's a reasonable outcome for a side project running at 20 cents a month. Understanding what actually drives traffic will take more data and more time.

## What's Next

The current iteration is a working foundation. Next priorities:

- **Better post quality for quantum/AMO articles** — the agent tends to write generic posts about "quantum physics" when the actual articles describe specific cold-atom experiments. The LLM only sees the page title and meta description; that's not enough context for scientific specificity.
- **Engagement metrics in the approval UI** — likes, boosts, and replies are returned by the platform APIs but not currently surfaced in the review interface.
- **Migrating to Mistral** — a straightforward model swap to test quality and cost.
- **Engagement with other accounts** — automatically liking relevant posts, replying to mentions with a draft that goes through the same approval flow.

The code is open source. If you're building something similar, feel free to take what's useful.
